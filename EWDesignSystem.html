Improving

Internal Design System

Overview

Client: Evolving Web
Project type: Internal design system 
Role: Sole designer
Timeline: 1 month ~ Q1 2023

When I started working at Evolving Web, they had just transferred their design system from Sketch to Figma. Overtime, I noticed their design process could become more efficient if they fully utilized Figma’s features, so I volunteered to improve it.

Balancing designer, developer, and business needs, I revamped the internal design system increasing designer, developer and QA productivity.

context

Less time pushing pixels, more time solving problems

The problem and objective

Designers were wasting time doing repetitive tasks because the Wireframe and Design Template—from which every project originates—didn’t take full advantage of Figma’s features. Documentation was tedious, application of components and spacing was inconsistent, and the design to developer and QA handoff wasn’t as clear as it could be.

So my goals involved improving the design system so that menial tasks are automated and/or expeditious, and the organization of the file is easier for developers and QAs to understand.

Discover

An origin file’s origin story

Primary research and insights

I took note of inefficiencies in the design process, and discussed them with the design team in feedback sessions. Additionally, a meeting was held with designers and developers to understand both perspectives.

Documentation was a time sink

Labeling and organizing components in Styleguide & Patterns (to handoff to devs and QAs) took too long and was tedious.

Components were lacking

Due to budget constraints, certain projects lacked mobile and/or tablet page mockups, forcing developers to make design choices that resulted in inconsistent design quality.

Reorganization and better navigation was necessary

Sometimes developers and QAs had trouble finding what they were looking for in Styleguide & Patterns.

More standardization was needed

Devs were frustrated with inconsistent spacing and pixel values that didn’t follow any logic.

Devs were onto something

Devs were experimenting with a Figma plugin called Tokens Studio that allows syncing of design tokens from Figma to code, speeding up frontend development.

Define

Balancing complexity is an art; too little and it's oversimplified, too much and it's overwhelming.

Goals

Given the pain points of both designers and developers, I decided on three main goals. With the fast pace, and varying needs, of projects at a design agency, I was mindful not to add more complexity than was needed because that would make it difficult to adopt for current and new teammates.

Improve the efficiency of the design process by fully utilizing Figma’s features and Figma Tokens.

Improve Wireframe and Design Template so that mobile and tablet mockups can always be made even with a small budget.

Eliminate confusion for developers and QAs when handing off designs.

Develop & Deliver

A design system that works for you, not the other way around

Video tutorials

To ensure the new system was simple to grasp and adoptable, I created video tutorials that covered all the updates and uploaded them to the agency’s intranet for future reference. I included notes from the videos too to ensure the information was discoverable.

Legible documentation

Separate text styles were made for documentation so that they won’t be affected when UI text styles change. Label components speed up the documentation process, and purple note components enable communication between designers, developers and clients. They’re used to leave final notes on how an element should function, while Figma comments are used for discussions. Please turn on your volume for all videos to listen to the rationale.

Hidden grids

Hidden grids on the component page show designers how components should be placed on the grid. It answers questions like “Does this span the full width of the screen, or does it stop at the margins?”

Responsive components

Components were made responsive to adjust to changes in component dimensions and screen sizes, and tablet variants were added so developers don’t have to make design decisions anymore. Even with a small budget, designers can duplicate desktop prototypes and change the variants to produce mobile and tablet prototypes in minutes.

Style tokens

A Figma plugin called Tokens Studio allows designers to apply, manage, and sync design tokens from Figma to code exponentially speeding up frontend development. 

It’s useful for designers too because Figma styles are limited to text and colours, but tokens allow global changes to shadows, border radius, border width, spacing and so much more. Imagine instead of clicking into every variant to adjust a style, designers can simply change the token value, and have it adjust across the entire document, page, or selection.

Spacing Tokens

Developers voiced their frustrations with inconsistent spacing between elements, so I suggested we adhere to the widely used 8-point grid system, and I made spacing tokens to ensure consistency. So imagine instead of going page by page to adjust spacing, designers can simply change the values for each use case in seconds.

Automatic documentation

I linked text documentation to tokens so they’d automatically update when the values change. For example, designers no longer need to label hex codes or font styles since design tokens will automatically update them.

Styleguide and patterns navigation

Styleguides & Patterns is a prototyped page in Figma that we send to devs and QAs. It covers everything from font styles to grid layouts to components used.

The main navigation was redone and better organized to match the atomic design system that the agency uses, which makes it easier for devs and QAs to find what they’re looking for.

Styleguide and patterns documentation

Before, components were laid out manually. Everytime, more components were added, we had to shift everything else on the page one by one. Now, components are organized with autolayout which means we can shift them using arrow keys, or pop them in between other components while maintaining the spacing. The documentation itself has options to add colour options, text, and icons, which the devs are familiar with.

Outcomes

Miscommunication with the UI team

In the process of experimenting, the UI team broke the link between tokens and main components. To resolve the miscommunication, I had to reiterate in a Q&A session that experimentation during the art direction phase should be done with instances instead of main components which were already linked to tokens.

Tokens are best for large-scale projects

We concluded that if a project only requires a few pages, tokens may not be necessary.

Praised for it's use of autolayout, responsiveness, and clarity

Because components were responsive and made to fit all three devices, and autolayout and label components were utilized, prototyping and documentation was more efficient and less overwhelming.

A design system devs love

Tokens sped up the deployment and updating of simple frontend code allowing devs more time to write creative code..
